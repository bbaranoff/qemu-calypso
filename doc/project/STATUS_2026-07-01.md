## TL;DR

Le vrai DSP c54x tourne, reçoit une **vraie tonalité FCCH** (FFT : pic +Fs/4 net dans
l'I/Q livré) et la consomme (PORTR @0x2a16). Mais il reste bloqué dans la **wait-loop
go-live** et n'arme jamais son IMR → l'ISR frame (corrélateur FCCH) ne tourne jamais →
`d_fb_det=0` → `FBSB_CONF=FAIL`. **Le mur n'est PAS un bug de décode** (tous corrigés) :
c'est un **handshake ARM→DSP jamais asserté**.

## Le mur, chaîne complète (désassemblé + tracé live)

```
wait-loop  0xa4d4 : BITF data[0x3f70], #0x0002     ; teste bit1 → lit 0x0000 → BOUCLE (0xa4e2: B 0xa4ca)
   └─ bit1 posé par le setter ROM ~0xde9c  ─ TC-gaté sur API 0x0314/0x0318 (= DSP words 0x098a/0x098c)
        └─ l'ARM n'écrit QUE 0x0000 dans ces cellules   (HS-ARM-GATE, calypso_trx.c ~553-566, read-only)
             └─ car le notifieur DSP→ARM api_write_cb est DÉCLARÉ mais JAMAIS assigné
                (calypso_c54x.h:204 ; appelé calypso_c54x.c:3357-3358 ; grep "api_write_cb *=" = 0)
                → l'ARM n'apprend jamais le boot-done DSP → n'asserte jamais l'enable
```

Go-live jamais atteint (désassemblé depuis le dump ROM) :
`0xa4c7 ORM #0x3000,IMR` → `0xa510/0xa518 ORM …,IMR` → `0xa51b RSBX INTM` →
`0xa51c LD *(0x08d4)` → `0xa51e STL A,*(0x3fb0)` → `0xa582 STL A,IMR (=0x52fd, vec28/bit12)`.
IMR effacé au boot par `0xb37e STM #0,IMR` (insn≈1047), **jamais ré-armé** (IMR=0x0000 tout le run).

# ADDENDUM 2026-07-01 (soir) — `calypso_arm2dsp.c` + franchissement go-live + direction 2

## Percée : le mur go-live est franchi, le corrélateur FB tourne

Nouveau module **`hw/arm/calypso/calypso_arm2dsp.c`** (+ `.h`) — orchestration ARM→DSP.
Câblé : `meson.build` (source), hook `calypso_arm2dsp_on_arm_write()` dans `calypso_trx.c`
(chemin write API-RAM), hook `calypso_arm2dsp_on_dsp_step()` dans `calypso_c54x.c` (boucle
exec). Capte l'événement d'orchestration ARM (`d_dsp_page` bit1 = B_GSM_TASK) et drive le DSP.

**Résultats obtenus (jamais atteints avant dans le projet) :**
- Premier `INTM=0`, première écriture `d_fb_det`, et **corrélateur FB atteint `PC=0x9ac0`**
  (noyau `0x9a80-0x9ac0`) — en drivant vers le bloc d'armement `0xa504`.
- **IMR bit3 armé et tenu** (`IMR=0x0008`) via arm-only mode.

## Ce que le drive chirurgical révèle (bug d'input corrélateur)

En drivant vers `0xa504`, le corrélateur tourne MAIS lit `[daram 0x2c00]=0x0001` constant,
**`AR3=0x2c00` figé** (ne balaie pas le burst), **aucun `IQ-READ`** → il MAC une cellule-flag
→ pas de pic → `d_fb_det=0`. Cause : le jump chirurgical **saute le setup de la frame-ISR**
(DMA du burst + pose AR3). Changer `CALYPSO_BSP_DARAM_ADDR` n'aide pas (pas un problème de où
on écrit, mais que le DMA vers la page corrélateur n'a pas lieu).

## Direction 2 (INT3 native) — mécanique complète implémentée

Pour que l'ISR frame monte le contexte proprement, il faut la **vraie IT INT3** (le hardware
empile le contexte). Implémenté :
- **`CALYPSO_C54X_IRQ_LEVEL=1`** (`calypso_c54x.c`, helper `c54x_irq_level_check` en tête de
  `c54x_exec_one`) : le modèle ne servait l'IT qu'au **edge** (appel `c54x_interrupt_ex`) — une
  IFR bit latchée pendant INTM=1 n'était jamais reprise. Ce check restaure le **niveau par
  instruction** (fidèle silicium). Mapping décodé : **vec = imr_bit + 16** (INT3 bit3→vec19).
- **`CALYPSO_BSP_INT3_UNGATED=1`** (`calypso_bsp.c`, 2 sites) : lève INT3 sans le gate
  `!(ifr&bit3)`.
- **`CALYPSO_DSP_IDLE_FF=0`** : sinon le DSP fast-forward sa boucle idle → `c54x_exec_one`
  sautée → le check niveau ne tourne jamais.
- arm2dsp arm-only : `CALYPSO_ARM2DSP_NOREDIR=1 IMR=0x08 ENAI=1` (arme IMR bit3 + clear INTM
  à la wait-loop 0xa4d4).

## Chaîne de 6 blocages couplés (état)

1. ✅ go-live (bit1 0x3f70 / handshake 0x098a-c) — contourné par arm2dsp.
2. ✅ armement IMR (0xa500 gaté 0x435b) — atteint (INTM=0, corrélateur).
3. ⏳ **contexte corrélateur** (DMA+AR3) — nécessite l'ISR native, pas un jump.
4. ✅ IT servie au niveau (IRQ-LEVEL) — implémenté.
5. ✅ gate BSP INT3 — ungated.
6. ⛔ **BLOCAGE COURANT : FN-matching / tagging FN des bursts**. `INT3-RATE=0` : la delivery
   `calypso_bsp_deliver_buffered` (1107) → `bsp_take_for_fn` (302) renvoie NULL car les bursts
   arrivent avec **`fn=0`** (non tagués par osmo-trx-ipc). `DL FN-LOCK dl_fnoff=581` est un
   artefact (calé sur un burst stale `fn=0`) → aucun offset fixe ne matche `burst_fn=0` figé
   contre `current_fn` mobile. `CALYPSO_DL_FN_OFFSET=581` ne débloque pas.

→ **Prochain amont** : pourquoi les bursts BSP ont `fn=0` (tagging FN côté ipc-device /
osmo-trx-ipc). Une fois tagués correctement → `bsp_take_for_fn` matche → delivery → INT3
(ungated) → IRQ-LEVEL sert l'IT → ISR native (DMA+AR3+contexte) → corrélateur sur vrai I/Q.
⚠️ Reste aussi le point **IPTR** : INT3 à IPTR=0x1ff → stub 0xffcc ; il faut IPTR=0x140 → 0xa04c.

# ADDENDUM 2 — 2026-07-01 (nuit) : direction 2 native prouvée, mur unique isolé

## La mécanique IT native fonctionne intégralement (prouvé)

Chaîne complète implémentée et **vérifiée en live** :
- BSP lève la frame IT (`c54x_interrupt_ex(19,3)`), delivery `deliver_buffered` (`running=1`,
  `ungated=1`, `delivered` monte).
- Remap `CALYPSO_DSP_FRAME_VEC28` : vec19/bit3 → **vec28/bit12** (d'où `IFR=0x1000`).
- **Découverte clé** : il fallait armer `CALYPSO_ARM2DSP_IMR=0x1000` (bit12) pour matcher le
  remap — sinon `ifr(0x1000) & imr(0x08) = 0`, le check niveau ne servait rien (mismatch vecteur).
- **`CALYPSO_C54X_IRQ_LEVEL` sert enfin l'IT** : `IRQ-LEVEL take bit=12 vec=28 -> PC=0x00f0
  IPTR=0x001` (60×). IPTR correct (0x001×0x80 + 28×4 = 0xf0 → `data[0xf0]` branche 0x7234, PAS
  le stub 0x1ff). En vec19 : `bit=3 vec=19 -> PC=0x00cc`.

## Le mur UNIQUE restant : contexte go-live de l'ISR

**Quel que soit le vecteur (vec19 ET vec28) et avec/sans `SEED5AC8`, l'ISR DÉRAILLE** :
`POST-BOOTSTUB-RET PC=0x0000` (storm 156-216). Le corrélateur (`0x9xxx`) n'est jamais atteint,
le DSP reste dans la wait-loop `0xa4ca..0xa4e2`.

**Cause, définitive** : on déclenche l'IT frame pendant que le DSP est **pré-go-live** — wait-loop,
**SP=0x5ac8** (stack de boot), sans le contexte/registres que le bootstrap ROM pose. L'ISR frame
attend l'état **post-go-live** (SP=0x1100, IMR=0x52fd, frame poussée) ; son épilogue POPM/RET
dépile du néant → PC=0. `SEED5AC8` (seed mem[0x5ac8]=0x71f4) ne suffit pas (contexte plus profond
qu'une seule cellule stack).

**C'est le MÊME mur que le drive chirurgical** (0xa504 atteint le corrélateur mais sans DMA/AR3) :
les deux voies butent sur **l'absence d'un état DSP go-live cohérent**, que seul le vrai bootstrap
ROM établit — et qui est bloqué au mur go-live d'origine (handshake 0x098a/0x098c jamais armé).

## Où on en est (problème réduit à UNE question)

> **Amener le DSP à un état go-live cohérent (SP=0x1100 + contexte ISR) pour que la frame-ISR
> native tourne (DMA + AR3 + corrélateur).** Tout le reste est câblé et prouvé.

Deux pistes pour une future session :
1. **Compléter le vrai bootstrap** : débloquer le handshake go-live d'origine (l'ARM écrit
   non-nul en 0x0314/0x0318 → setter 0xde9c pose 0x3f70 bit1 → le ROM atteint 0xa582, arme
   IMR=0x52fd, pose SON contexte) → ensuite l'IT native tourne sur un état sain. **La voie fidèle.**
2. **Synthétiser le contexte ISR complet** avant de vectoriser (SP=0x1100 + frame de registres
   que l'épilogue POPM attend) — plus fragile, mais teste l'ISR isolément.

# ADDENDUM 3 — 2026-07-01 (nuit tard) : le handshake go-live = état-machine 5-cellules

## Résumé décisionnel

La voie fidèle pour amener le DSP en post-go-live = **le vrai handshake `0x098a..0x098e`**.
Ce n'est PAS 2 cellules (0x098a/0x098c) ni un simple "non-nul" : c'est un **état-machine ROM
à 5 cellules** (`data[0x098a..0x098e]`, code `0xdde0-0xde9f`) qui **calcule un index de dispatch**
routant soit vers le reclear `0xde8b` (bit0), soit vers le setter `0xde9c` (`ST #0x0002,0x3f70`
= bit1 = sortie wait-loop → go-live). Atteindre `0xde9c` demande la **combinaison exacte** des
5 valeurs. Des valeurs uniformes (0x0001, 0xFFFF sur les 5) → NE routent PAS vers le setter.

## Décode partiel du flot (0xdde0 →)

Entrée setter `0xdde0` (atteinte en live via `0xdddb→0xddde→0xdde0`) :
```
0xdde8 : (op 0x71 sur 098a/098c)
0xddeb : LD  *(0x098a), A
0xdded : BC  0xde8a  si A==0        ; 0x098a==0 → 0xde8a → 0xde8b RECLEAR (bit0)
0xddef : LD  *(0x3f63), B ; ...
0xddf3 : B   0xde86
0xde86 : LD  *(0x098c), A
0xde88 : BC  0xddf5  si A!=0        ; 0x098c!=0 → 0xddf5
0xddfd : LD  *(0x098d), A
0xddff : BC  0xde82  si A==0        ; 0x098d==0 → 0xde82 → BOUCLE 0xde86↔0xddf5↔0xde82
0xde01 : LD  *(0x098b), A ; 0xde03 BC 0xde32 si 098b==0
0xde09 : LD  *(0x098d), A ; 0xde0b BC 0xde28 si 098d==0
0xde0e : ADD *(0x098b), A ; 0xde10 BC 0xde1f si A==0
...     (ADD/BITF/SFTA sur 098b/098c/098d/098e → calcul d'index)
setter : 0xde94 (71f8 098a 098c) → 0xde97 BITF 0x3f70 #1 → 0xde9a BC deab si NTC
         → 0xde9c ST #0x0002, 0x3f70   (bit1 — LE BUT)
```
Table de dispatch (adresses cibles) à `~0xde27` : entrées `de87/de8b`(reclear)…`de9c`(setter),
`de9c` en haut de table = index haut.

# ADDENDUM 20 — [2026-07-03] BOUCLE FERMEE : le point d entree IMR-arming existe, jamais atteint

## LA DECOUVERTE QUI FERME LE DOSSIER
Decode precis de 0xa4c0-0xa4cd (PROM0, confirme octet-exact) :

0xa4c3: ST  #0, data[0x3f92]

**`0xa4c7` (l ORM d armement) : 0 hits sur toute la session, TOUS runs.**
**`0xa4ca` (3 mots plus loin, meme "boucle") : 130 hits, l entree TOUJOURS empruntee.**
`fc00`(RET) juste avant 0xa4c7 confirme que ce n est PAS un fall-through normal — 0xa4c7
est un point d entree SEPARE (jump target), pas une suite sequentielle de ce qui precede.

## Connexion avec addendum 15 (jump table / dispatcher CALA)

dispatch a 0xaae7-0xab37 ... 2 entrees pointant vers 0xa4c7"*. **C EST LE MEME MECANISME.**

re-ecrire sa propre valeur courante -> resout TOUJOURS vers le stub no-op `0xab38` au lieu

## Discriminant utilisateur (ISA precis) : verdict final
- Hypothese B (mis-decode dual-operand, precedent 0x8a46/SP) : REFUTEE pour 0xb37e
  (`STM #0,IMR` explicite et legitime, addendum 19).
- Hypothese A (liveness, re-arm jamais atteint) : **CONFIRMEE ET LOCALISEE PRECISEMENT.**
  Le re-arm existe (`ORM #0x3000,IMR @0xa4c7`), il est cible par la table de dispatch, mais
