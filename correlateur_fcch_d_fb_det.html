Corrélateur FCCH — le mur d_fb_det = 0

Cause décode-attribuable, sur la voie autoritaire : case 0x5 mis-route la famille DADST/DSADT (dual long-word) en SFTA/SFTL. La somme-de-produits de corrélation n'est jamais calculée → l'accumulateur publié est figé → d_fb_det reste à 0, donc pas de SB, pas de SI.

Ce que le bug détruit

case 0x5 = SFTA/SFTL only · lignes 7817–7852
  • la lecture du Lmem 32-bit à *AR5jamais lue comme opérande
  • le ± T (dual add/sub) — le cœur du calculremplacé par « A >>= ASM(-3) »
  • le post-mod *AR5+ (stride dans le burst)droppé · walked = 0 / 40
  • le wrap circulaire *AR5+0% (BK=0x20)droppé
⇒ A n'est que décalé (shiftLike=1), B jamais touché (dB=0) : 0 corrélation.

Recette du handlergaté c54x · shunt-safe
// intercepter 0x5A/5B/5E/5F (et idéalement 0x50-59) AVANT le bloc shift
// décoder Lmem via resolve_smem (+ c54x_circ_ref pour BK)
switch (s->st1 & ST1_C16) {

  case C16 = 1:  // dual-16
      dst(39-16) = Lmem(31-16) ± T ;
      dst(15-0)  = Lmem(15-0)  ∓ T ;

  case C16 = 0:  // double-precision
      dst = Lmem ± ((T << 16) + T) ;
}
// DADST = (+ , −)   DSADT = (− , +)
// dst = bit8 → A / B
Note importante. case 0x5 mis-route en réalité toute la plage 0x50-0x5F (7 instructions dual long-word), pas seulement DADST/DSADT. Le noyau FCCH n'utilise que DADST/DSADT, mais un fix propre doit repartitionner proprement — et vérifier qu'aucun vrai SFTA/SFTL ne vit réellement dans 0x5x. La doc dit que non (les shifts sont ailleurs) ; à confirmer avant de toucher.
